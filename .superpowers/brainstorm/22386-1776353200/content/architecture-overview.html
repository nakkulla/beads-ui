Worker Jobs: 현재 구조 vs 추천 구조

추천안은 Backend/Supervisor 분리입니다. 핵심은 job의 truth source를 UI 서버 밖으로 분리하는 것입니다.

현재 구조
Flow
Browser UI
beads-ui server
API + in-memory jobs + spawn(codex exec)
child process

장점

  • 구현 단순
  • 기존 구조 확장 쉬움

한계

  • 서버 재시작 시 상태 취약
  • cancel/log/recovery 확장성 낮음
  • UI 서버에 책임 집중
추천 구조 (Approach B)
Flow
Browser UI
beads-ui server
UI + API gateway
Local Job Backend / Supervisor
job store + runner + cancel + recovery
Runner: direct process
Optional attach/debug: tmux
logs / events / elapsed time

장점

  • 서버 재시작 후에도 복구 가능
  • cancel, log, history 확장 쉬움
  • UI와 실행 책임 분리

비용

  • 구성요소 1개 증가
  • 초기 설계 난이도 상승
A

이 방향 유지

Supervisor 분리 구조로 설계를 계속 진행합니다.

B

조정 필요

분리 구조는 맞지만, 경계나 구성요소를 더 단순화/변경하고 싶습니다.
